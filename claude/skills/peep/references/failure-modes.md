# Failure Modes Catalog

This is the empirical catalog of failure modes that peep templates guard against. Each mode has a short definition, the symptom you'll see, the template section that catches it, and a citation. Read this when you want to understand *why* a template section exists or when you're tempted to skip one.

## Universal failure modes (all templates) — Ugare & Chandra (2026)

These three modes were identified by error analysis on Opus-4.5 + semi-formal reasoning across patch equivalence, fault localization, and code QA tasks.

### U1. Incomplete execution tracing
**Symptom**: agent assumes function behavior without tracing the actual call chain. Says "format() returns the formatted string" without grepping for `def format`.
**Caught by**: every template's mandatory `file:line` citation rule. Function trace tables in code-qa.md. Code path tracing in fault-localization.md.
**Source**: Ugare & Chandra, *Agentic Code Reasoning*, Meta 2026, §4.1.2 ("Error Analysis for Verifier With Tests").

### U2. Third-party library semantics
**Symptom**: agent guesses what `library.function()` does from its name. "redis.set() persists the value" without checking whether persistence requires `SAVE`.
**Caught by**: explicit `ASSUMPTION: ... Source not verified` flag required in universal rules of engagement (SKILL.md).
**Source**: Ugare & Chandra 2026, §4.1.2.

### U3. Dismissing subtle differences
**Symptom**: agent identifies a semantic difference but concludes "it doesn't matter for the test". Often wrong — the difference does matter under one input the agent didn't trace.
**Caught by**: counterexample-or-proof requirement; differential check sections.
**Source**: Ugare & Chandra 2026, §4.1.2.

## Fault localization specific

### F1. Stopping at the crash site
**Symptom**: `StackOverflowError` at line 185 → agent reports line 185 as the bug. Real bug is upstream at line 80 where the recursive condition was set up.
**Caught by**: PHASE 4 `DIFFERENTIAL CHECK` in fault-localization.md ("if the bug were at the crash site rather than the predicted root cause, what should we observe that we don't?").
**Source**: Ugare & Chandra 2026, Appendix C (Mockito_8 case study).

### F2. Indirection bugs
**Symptom**: Test calls `CSVParser.parse()` and the bug is in `CSVFormat.withHeader()`. Agent fixates on the directly-invoked class.
**Caught by**: hypothesis-driven exploration loop with H1 ("bug is in the directly-invoked class") that gets REFUTED, freeing the search.
**Source**: Ugare & Chandra 2026, §4.2 (error analysis on 90-bug Defects4J sample, Csv_12).

### F3. Multi-file bugs
**Symptom**: bug spans 2+ files; agent finds one location and stops.
**Caught by**: ranked predictions section requires multiple regions; differential check forces reconsideration.
**Source**: Ugare & Chandra 2026, §4.2.

### F4. Domain-specific bugs (numerical, concurrency)
**Symptom**: bug requires specialized knowledge (e.g., eigenvalue decomposition stability); pattern-matching fails.
**Caught by**: explicit `ASSUMPTION` flag; differential check pushes back on "obvious" answers.
**Source**: Ugare & Chandra 2026, §4.2 (Math_81 EigenDecomposition case).

## Code QA specific

### Q1. Long reasoning chain that misses one step
**Symptom**: agent traces 5 calls correctly, then misses that downstream code already handles the edge case it identified. Confident-but-wrong elaboration.
**Caught by**: ALTERNATIVE HYPOTHESIS CHECK in code-qa.md.
**Source**: Ugare & Chandra 2026, Appendix D (py_5 example).

### Q2. Guessing API differences from naming
**Symptom**: asked whether `map.at(key)` differs from `map[key]`, agent invents non-existent edge cases ("if an invalid key is somehow passed...") instead of checking where keys are actually generated.
**Caught by**: function trace table + DATA FLOW ANALYSIS sections force tracing the actual key sources.
**Source**: Ugare & Chandra 2026, Appendix D (cpp_3 example).

## Brownfield construction specific

### B1. Pattern reinvention
**Symptom**: agent builds a parallel implementation of `retry_with_backoff` while one already lives in `utils/retry.py`.
**Caught by**: EXISTING PATTERN SURVEY mandatory table in brownfield-construction.md.
**Source**: AgentPatterns.ai, *Pattern Replication Risk in Agentic Code Generation* — combined with the inverse documented in the next mode.

### B2. Deprecated-pattern reuse
**Symptom**: agent finds `legacy_auth_check()` in the repo and reuses it, not knowing the team replaced it with `AuthMiddleware` 6 months ago. The legacy function is still there because removal is a separate ticket.
**Caught by**: "Still preferred?" column in the survey table forces explicit ask, not just existence check.
**Source**: AgentPatterns.ai, *Pattern Replication Risk*; arxiv 2407.06153v2 *What's Wrong with Your Code Generated by Large Language Models?* (taxonomy of generation issues including deprecated-API use).

### B3. Invariant violation
**Symptom**: change passes new tests but silently breaks an implicit assumption (e.g., "every User row has a non-null tenant_id" — relied on by 12 callers, asserted nowhere).
**Caught by**: INVARIANTS THAT MUST BE PRESERVED section requires explicit identification with file:line evidence.
**Source**: arxiv 2603.27524 (agents introduce breaking changes during refactoring/maintenance at 6.72%–9.35% rates).

### B4. Scope creep / over-refinement
**Symptom**: change touches 12 files when 3 would have sufficed; "improving while I'm here". Formal name from refinement calculus: *over-refinement* (Sciencedirect 2002, *Requirements, specifications, and minimal refinement*).
**Caught by**: D2 (MINIMAL) — every CHANGE SURFACE row must cite an Rn.
**Source**: refinement calculus literature (see `refinement-vocabulary.md`); Anthropic's *Simplicity First* coding guidelines.

### B5. Spec drift
**Symptom**: agent builds what it imagined ("a search feature") instead of what was asked ("filter the existing list by status"). The agent's mental model of the requirement diverged from the user's words.
**Caught by**: SPEC verbatim + atomic Rn decomposition; D1 (SUFFICIENT) requires every Rn maps to code.
**Source**: arxiv 2602.17037 (taxonomy of agent misbehaviors at scale, including "Specification Drift" as a primary category).

### B6. Integration mismatch
**Symptom**: new code has a beautiful API but doesn't connect to the actual callers; the function expects a `User` object and callers pass a `dict`.
**Caught by**: INTEGRATION POINTS table + BACKWARD-COMPATIBILITY TRACE for existing callers.
**Source**: arxiv 2603.27524 (regression in untouched code); Kurten 2026, *75% of AI Agents Break Code They Maintain* (~75% of agents introduce regressions per maintenance loop).

### B7. Regression in existing tests
**Symptom**: new tests pass; agent doesn't run the existing PASS_TO_PASS suite; CI catches the breakage hours later.
**Caught by**: BACKWARD-COMPATIBILITY TRACE → "Existing tests touching the change surface".
**Source**: angelkurten.com / Kurten 2026; arxiv 2603.27524.

## Greenfield construction specific

### G1. Premature abstraction
**Symptom**: a 30-line CSV-to-JSON CLI grows into `ApplicationService`, `CommandBus`, `Repository`, `SerializerFactory`, `PluginRegistry`. Each class has one implementation.
**Caught by**: ADVERSARIAL CHECK question 1 in greenfield-construction.md ("is any module/class/interface only used in one place?"); CREATED SURFACE table requires Rn justification per row.
**Source**: greenfield agent assessment by Hephaestus (gpt-5.5, May 2026), `v2-greenfield-assessment.md`.

### G2. Unjustified dependency choice
**Symptom**: a static landing page is scaffolded with SSR + ORM + auth + queues because no existing pattern argues against it.
**Caught by**: dependencies table requires "Could we do without?" column; ADVERSARIAL CHECK question 2 (~50-line threshold).
**Source**: Hephaestus assessment, May 2026.

### G3. No runnable verification path
**Symptom**: agent writes `test_*.py` files but no `package.json` script, no `pytest` config, no fixtures. A fresh clone cannot run anything.
**Caught by**: VERIFICATION SCAFFOLD section is mandatory and requires concrete commands; D4 (VERIFIABLE) cannot be satisfied without it.
**Source**: Hephaestus assessment, May 2026 — specifically called out as the failure mode brownfield-only design would miss.

### G4. Framework-of-the-week
**Symptom**: agent picks the trendy library (e.g., today's hot ORM) without articulating why it beats older stable alternatives or no library at all.
**Caught by**: DESIGN OPTIONS CONSIDERED requires at least 2 alternatives with rejection reasons.
**Source**: Anthropic *Simplicity First* guideline (CLAUDE.md); Hephaestus assessment.

### G5. Untestable boundaries
**Symptom**: code structured so the only way to test is end-to-end (network + DB + LLM). Inner-loop testing impossible.
**Caught by**: ADVERSARIAL CHECK question 3 ("can NT1 be run with NO network, NO real DB, NO LLM?").
**Source**: greenfield assessment + standard testability literature.

### G6. Invented future requirements
**Symptom**: spec says "auth for one user role"; agent builds plugin-loader, role-system, multi-tenancy, audit log. None requested.
**Caught by**: SPEC DECOMPOSITION + non-goals; D2 (MINIMAL) — each created surface row must cite a stated Rn or Cn.
**Source**: Anthropic CLAUDE.md *Simplicity First*; arxiv 2602.17037 (taxonomy includes overgeneration).

## How to use this catalog

When a template tells you to "guard against" a mode, find the entry here. If you're tempted to skip a section because "it's obviously fine in this case", look up which mode that section guards against — the empirical rate at which agents fall into it is rarely zero.
