# Citations

Bibliography for the peep skill — the literature behind each design choice and failure-mode catalog entry. Open this when you need to cite a source in a report or want to follow up.

## Primary source — semi-formal reasoning

- **Ugare, S. & Chandra, S.** (2026). *Agentic Code Reasoning*. Meta. arXiv:2603.01896v2.
  The paper that introduces semi-formal certificate-based reasoning for code analysis tasks. Reports +5–12pp accuracy gains on patch equivalence verification (88% vs 78%, with 93% on real-world patches), code question answering (87% vs 78%), and fault localization (Top-5: +5–12pp on Defects4J). Full source is in `peep.pdf` alongside this skill.

## Construction-side extensions (peep v2)

The construction templates extend the paper's analysis-only frame into design / planning. The framework is an informal version of correct-by-construction.

- **Back, R.-J. & von Wright, J.** (1998). *Refinement Calculus: A Systematic Introduction*. Springer.
  Foundation for the SUFFICIENT / MINIMAL / over-refinement vocabulary used in the construction templates.

- *Requirements, specifications, and minimal refinement* (2002). *Sciencedirect S1571066104805504*.
  Origin of the "over-refinement" concept used in D2 of construction templates.

- **Dijkstra, E. W.** (1976). *A Discipline of Programming*. Prentice-Hall.
  Predicate transformer semantics (wp / sp) cited in `refinement-vocabulary.md`.

- **Meyer, B.** (1988). *Object-Oriented Software Construction*. Prentice-Hall.
  Design-by-contract (Pre / Post / Inv) used in INITIAL CONTRACTS AND INVARIANTS sections.

- **Chlipala, A.** (2014). *Fiat: Deductive Synthesis of Abstract Data Types in a Proof Assistant*. MIT/POPL.
  Reference architecture for deductive synthesis cited as the inspiration for the informal CbC template style.

- **Polikarpova, N. et al.** (2016). *Program synthesis from polymorphic refinement types* (Synquid). PLDI.
  Spec-driven synthesis with refinement types.

- **Kasibatla, S. R. et al.** (2025). *Cobblestone: A Divide-and-Conquer Approach to Automating Formal Verification*. arXiv:2410.19940.

## Inference-rule taxonomy (claim types)

- **PCRLLM**: *Proof-Carrying Reasoning with LLMs under Stepwise Logical Constraints*. arXiv:2511.08392.
  Direct prior art for the "every claim cites the inference rule it uses" pattern in `claim-types.md`.

- **ProofSketcher**. arXiv:2604.06401.
  Hybrid design where each step has an explicit type (rewrite, case split, induction, contradiction). Inspiration for the eight-category taxonomy in `claim-types.md`.

## Hypothesis-driven exploration

- **HypoExplore (Wang et al.)** (2026). *Agentic Discovery with Active Hypothesis Exploration for Visual Recognition*. arXiv:2604.12999.
  Source of the explicit confidence thresholds (c≥0.75 confirm, c≤0.25 refute) and the trajectory-tree + memory-bank pattern in `exploration-loop.md`.

- **Yang, J. et al.** (2024). *SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering*.
  The minimal SWE-agent setup used in the Ugare & Chandra paper for agentic exploration.

- **CodeNav**: arxiv 2406.12276v1. Iterative search-execute-fix loop for codebase snippet discovery — alternative agentic exploration pattern.

## Brownfield failure modes

- **AgentPatterns.ai** — *Pattern Replication Risk in Agentic Code Generation*.
  Documents B1 (pattern reinvention) and especially B2 (deprecated-pattern reuse).

- **arxiv 2407.06153v2** — *What's Wrong with Your Code Generated by Large Language Models? An Extensive Study*.
  Taxonomy of LLM-generation issues including deprecated-API use, integration mismatch.

- **arxiv 2602.17037** — *Taxonomy of agent misbehaviors at scale*.
  Source of the "Specification Drift" category (B5).

- **arxiv 2603.27524** — Agent maintenance behavior at scale.
  Empirical 6.72% breaking-change rate for refactoring tasks; 9.35% for chore changes.

- **Kurten, A.** (2026). *75% of AI Agents Break Code They Maintain*. angelkurten.com.
  Empirical "75% of models break previously working code during maintenance iterations" — direct justification for the BACKWARD-COMPATIBILITY TRACE section's strictness.

## Greenfield failure modes

- **Hephaestus assessment** (May 2026). `v2-greenfield-assessment.md` (gpt-5.5 high adversarial review).
  Identified G1 (premature abstraction), G2 (unjustified dependency choice), G3 (no runnable verification path) as failure modes a brownfield-only design would not catch. Recommendation to split brownfield + greenfield templates is grounded here.

## Counterexample / sufficiency framing

- **Polytechnique 2019** — *Property-Based Testing via Proof Reconstruction*.
  Frames PBT as proof-theoretic. Used in the COUNTEREXAMPLE / SUFFICIENCY CHECK section's per-Rn binary (soundness sketch OR concrete counter-input).

- **FuzzChick** (lemonidas.github.io) — coverage-guided property-based testing.
  Inspiration for "which input partitions does NTn cover" prompt.

- **arxiv 2101.06825** — Counterexample-guided prophecy in model checking.

## Skill design — why nested files

- **Anthropic** — *Skill authoring best practices*. platform.claude.com / docs.anthropic.com.
- **Skill-creator** (Anthropic plugin). The official skill-creator at `~/.claude/plugins/cache/claude-plugins-official/skill-creator/` documents the three-level loading model (metadata always; SKILL.md on trigger; references/ on demand).
- **emergentmind.com**, *Chain-of-Thought Length in LLM Reasoning*; arxiv 2502.07266 — empirical inverted-U for CoT length; redundancy priors. Used to justify the lean SKILL.md router + on-demand references/ pattern.

## How to cite peep itself

If you write a report whose conclusions came out of filling a peep certificate, cite both the underlying paper and the certificate:

> Reasoning followed the semi-formal certificate methodology of Ugare & Chandra (2026, arXiv:2603.01896v2), using the [template-name] template from the peep skill.
